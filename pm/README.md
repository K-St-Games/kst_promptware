# PM Export

A tool-agnostic projection of this repo's governance artifacts into a single file — [`pm-export.json`](pm-export.json) — that a downstream adapter can push into GitHub Issues, Linear, Jira, or a custom tracker.

This is the repo's answer to *"we track work in docs, but eventually want it in a real PM tool"* — without committing to any one tool, and without a second source of truth.

## Source of truth

**Derived, not canonical.** The work orders, capability map, and ledger are the truth. This file is a snapshot regenerated from them:

- Regenerate with the [`pm-export`](../skills/pm-export/SKILL.md) skill — do **not** hand-edit item content.
- The one field the docs do not hold is **`external_ids`** — the id each tool assigns on first push. Regeneration preserves it (matched by canonical `id`); losing it orphans synced items and re-syncing duplicates them.
- **`source_hash`** on each item lets a pusher detect which items drifted since the last sync.
- Flow is one-way: **docs → JSON → tool.** State is never pulled back — that would lose the `reconciled` / `aborted` distinctions no tracker can represent.

## What's in it

| `kind` | Projected from | Example ids |
|---|---|---|
| `epic` | Capability-map feature rows | `F-001` … `F-016` |
| `task` | Work orders in `work-orders/` | `WO-0001` |
| `question` / `decision` | Ledger entries | `L-001` … `L-015` |

Canonical `status` values are the union of the work-order state machine (Constitution §8), the capability-map markers (`queued`, `delivered`), and the ledger statuses (`open`, `resolved`, `archived`). Target tools translate them via the mapping table in the skill — the export stays tool-neutral.

## Regenerating

1. Invoke `/pm-export` (or follow `skills/pm-export/SKILL.md`).
2. It reads `work-orders/`, `capability-map.md`, and `ledger.md`, reconciles against the existing export (preserving `external_ids`), and validates against the schema.
3. Review the emitted **added / changed / closed** diff before committing.

The first concrete push target is GitHub Issues; the neutral core is unchanged for Linear or Jira.
