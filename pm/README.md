# PM Export

A tool-agnostic projection of this repo's governance artifacts into a single file ‚Äî [`pm-export.json`](pm-export.json) ‚Äî that a downstream adapter can push into GitHub Issues, Linear, Jira, or a custom tracker.

This is the repo's answer to *"we track work in docs, but eventually want it in a real PM tool"* ‚Äî without committing to any one tool, and without a second source of truth.

**Governance status:** this README is the governed artifact for the `pm/` directory. `pm-export.json` itself is generated output ‚Äî it carries no governance header of its own (JSON has no natural place for one, and Constitution ¬ß2.2 already carves out an analogous exception for `/LICENSE`); it is governed *through* this README and the skill that produces it, not independently.

## Source of truth

**Derived, not canonical.** The work orders, capability map, and ledger are the truth. This file is a snapshot regenerated from them:

- Regenerate with the [`pm-export`](../skills/pm-export/SKILL.md) skill ‚Äî do **not** hand-edit item content.
- The one field the docs do not hold is **`sync`** ‚Äî a per-target checkpoint (`{"github": {"external_id": 42, "last_pushed_hash": "sha256:..."}}`) recording what each tool assigned on first push and what it last saw. Regeneration preserves it (matched by canonical `id`); losing it orphans synced items and re-syncing duplicates them.
- **`source_hash`** on each item, compared against `sync[target].last_pushed_hash`, lets a pusher detect which items drifted *for that specific target* since its last sync ‚Äî drift is tracked per target, not globally.
- An item whose source disappears from the docs but was previously synced becomes a **tombstone** (`present: false`) rather than being force-assigned a closing status ‚Äî it's dropped once every target that had it reports `closed: true`.
- Flow is one-way: **docs ‚Üí JSON ‚Üí tool.** State is never pulled back ‚Äî that would lose the `reconciled` / `aborted` distinctions no tracker can represent. (An adapter writing its own `last_pushed_hash` / `closed` after a successful push is bookkeeping about its own action, not an exception to this.)

## Regenerating

1. Invoke `/pm-export` (or follow `skills/pm-export/SKILL.md`).
2. It reads `work-orders/`, `capability-map.md`, and `ledger.md`, reconciles against the existing export (preserving `sync`, per-target), and validates against the schema plus the semantic checklist.
3. Review the emitted **added / changed / closed** diff (broken out per target) before committing.

The first concrete push target is GitHub Issues; the neutral core is unchanged for Linear or Jira.
