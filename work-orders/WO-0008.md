This order establishes the clean baseline needed before separate orders amend governance, reconcile the v3/v4 seam, or harden the PM export contract.

## 1. Goal

After this order completes, one dependency-free command mechanically checks the active documentation tree for resolvable local links, structurally valid agent and skill frontmatter, catalog/profile consistency, required governance headers on explicitly governed Markdown artifacts, and unresolved initialization placeholders. The known failures are repaired, the command is part of the closeout playbook, and the repository has a clean evidence-backed baseline for the three follow-up improvement orders.

## 2. Non-Goals

1. Ratifying a lightweight governance path, adding artifact classes to the charter, or changing the repository's product boundary. Those require architect and human approval through WO-0009.
2. Integrating or restructuring the v3 session layer, defining `T-###` to work-order status mapping, or introducing machine-readable installation manifests. Those belong to WO-0010.
3. Changing PM export sync-state, tombstone, status, hash, or idempotency contracts, or implementing a GitHub/Linear/Jira adapter. Those belong to WO-0011.
4. Adding a package manager, build system, standing service, network dependency, or third-party runtime package.
5. Repairing deprecated content under `/archive/`; the audit excludes it unless an active document links into it.

## 3. Reads, Discovery & Writes

All three lists are closed-world (fixed).

**Reads** provide the governing norms, current validation contracts, catalog/profile claims, session-layer boundary, and accepted review evidence. No proposal under `/references/` other than `pm-feature-feedback.md` may introduce a norm into this order.

**Discovery roots** allow exhaustive factual inspection of active Markdown links, artifact inventory, frontmatter shapes, and current catalog counts. Discovery may identify a mechanical conformance failure; it may not decide new governance behavior.

**Writes** are limited to the audit, its canonical documentation, the currently known broken active surfaces, work-order closeout artifacts, and the derived PM snapshot.

## 4. Constraints in Force

1. Constitution §§1, 2, 3, 5, 6, 7, 8, 9, and 10.
2. **Do not:** treat reference prose or discovery findings as authority for a new norm.
3. **Do not:** make the audit a canonical bundle dependency; it is an optional validation convenience.
4. **Do not:** silently weaken a check to make the current repository pass.
5. **Do not:** rewrite historical or archived documents solely to satisfy active-tree validation.
6. **Do not:** use model ids, remote service state, or network access as validation inputs.
7. **Do not:** resolve L-008, L-010, L-011, or L-015 by implication; route those decisions to their assigned follow-up orders.

## 5. Steps

Discretion rule (fixed): ambiguity about a norm or contract requires a ledger entry and escalation; ambiguity about error wording or report layout is the builder's discretion.

1. Define the audit's active-tree scope and exclusion rules directly from the reads and discovery roots. *(fixed)*
2. Implement `tools/docs-audit.mjs` with no third-party imports and stable nonzero exit behavior for failed checks. It must check local Markdown links, agent/skill frontmatter structure against the repository schemas' declared fields, catalog/profile inventory claims, four-line governance headers on Markdown files that declare an artifact class, and unresolved initialization tokens outside their defining templates. *(fixed)*
3. Add a deterministic `--self-test` mode covering at least a broken local link, malformed frontmatter with an unsafe unquoted colon, a stale catalog count, and an incomplete governance header. *(fixed)*
4. Run the audit and repair every failure whose correction is mechanical and lies within `writes`. Preserve the intended meaning of each repaired document. *(at discretion)*
5. Document the audit command, scope, and expected exit behavior in the validation playbook and canonical contributing guide. *(fixed)*
6. Run independent frontmatter/schema validation for this work order and the agent/skill catalogs, then run `git diff --check`. *(fixed)*
7. Refresh `pm/pm-export.json` from the final canonical sources while preserving any existing per-item `external_ids`; do not change the PM export contract in this order. *(fixed)*
8. At reconciliation, cut draft WO-0009, WO-0010, and WO-0011 with distinct parent features on the capability map; validate each draft but do not release or implement it. *(fixed)*
9. Present the audit output, validation evidence, and follow-up boundaries for the human `review -> complete` gate. *(fixed)*

## 7. Escalation Triggers

Standing triggers (fixed) — stop, set status `blocked`, and emit the escalation payload (§9); do not improvise — if:

1. A step requires mutating a surface outside `writes` beyond the transitive-write classes, or relying on context outside `reads` for a norm or contract.
2. A step requires a framework noun not resolvable by the constitution's glossary procedure.
3. The schema or a contract cannot express something this order needs.
4. An acceptance check fails for environment or infrastructure reasons; block as `infra_block` rather than entering the repair loop.
5. A discovered failure can only be repaired by amending the charter, constitution, artifact-class enumeration, mutation rights, or lightweight governance path.
6. Frontmatter validation requires accepting YAML behavior that cannot be implemented safely without a dependency or a narrower documented subset.
7. Correcting a catalog or profile claim would change the intended installation product rather than repair an objectively stale count or path.
8. PM export regeneration requires choosing among the unresolved sync-state, deletion, or idempotency designs assigned to WO-0011.
